## 1. 总体目标

把项目从“自建工具 + 本地模型优先”的旅游规划器，升级为：

- **云端模型优先**：首选阿里云百炼 `qwen3.6-plus`
- **本地模型兜底**：次选 `Ollama + qwen3:14b`
- **MCP 统一工具层**:
  - 地理能力：阿里云百炼 `Amap Maps MCP`
  - 金融能力：`盈米金融数据 MCP / OAP`
- **核心卖点升级**:
  - 旅游规划从“静态行程生成”变成“真实地理约束下的动态规划”
  - 新增一个和旅行强相关、但又能体现金融数据接入能力的功能模块

## 2. 模型选型决策

### 2.1 云端主模型：`qwen3.6-plus`
我建议把它作为**主编排模型**，原因很直接：

- 百炼官方 MCP 接入示例直接使用它
- 支持 `Responses API + MCP`
- 支持 `Function Calling / 结构化输出 / 内置工具`
- 1M 上下文，适合多轮规划、工具返回结果整合、复杂 itinerary 生成
- 阿里官方对复杂任务也更偏向 `Plus` 系列

**结论**:
`qwen3.6-plus` 作为主模型，比 `qwen-max` 更适合你这个“多工具、多约束、需要结构化落库”的项目。

### 2.2 降本可选子模型：`qwen3.6-flash`
不是替代主模型，而是给以下子任务用：

- 城市名 / 日期 / 预算 / 人群类型抽取
- 评论摘要
- 工具返回结果压缩
- 非关键解释文案生成

### 2.3 本地兜底模型：`gemma4:e4b`（Ollama）

## 3. 架构重构目标

### 3.1 最终结构

- **Java Spring Boot**
  - 主业务后端
  - 用户、任务、帖子、通知、排行榜、分析
  - AI 任务调度、结果落库、对前端出 API

- **Python Planner Service**
  - 不再自己做主要事实检索
  - 改为：
    - 调用百炼 `Responses API`
    - 挂载 MCP Server
    - 做多阶段规划编排
    - 输出结构化 JSON

- **MCP 工具层**
  - `Amap Maps MCP`：官方百炼云部署服务
  - `Yingmi Finance MCP`：优先直连；若对外仍只给 OpenAPI,则加一个内部 `yingmi-mcp-bridge`

### 3.2 关键设计原则

- **旅游事实以高德 MCP 返回为准**
- **模型负责推理，不负责编地理数据**
- **金融能力做“辅助决策信息”，不做投资建议引导**
- **所有 MCP 调用都要落 trace,便于面试展示可观测性**

你的 Python Planner 改为通过百炼 `Responses API` 挂 MCP 工具。

挂载两个工具源：

- Java:业务编排

## 5. 高德地图 MCP 驱动的旅游规划增强方案

这是这次升级的核心。

### 5.1 你原先旅游规划的问题
你现在的规划逻辑更偏：

- 先抽城市
- 再搜景点
- 再拼 itinerary

问题是：

- 缺少真实地理约束
- 景点排序不一定可执行
- 同一天可能跨城/绕路
- 餐饮/交通/天气联动不够强

### 5.2 新的规划链路
我建议改成 6 段式：

1. **意图解析**
   - 城市、出发点、天数、预算、兴趣、人群、交通偏好

2. **地理 grounding**
   - 用高德 MCP 做：
     - 起点/终点地理编码
     - 核心区域识别
     - POI 搜索
     - 周边设施搜索

3. **候选 POI 池构建**
   - 按主题分层：
     - 必去景点
     - 餐饮点
     - 休息点
     - 备选点
   - 每个点附带：
     - 地址
     - 坐标
     - 类别
     - 建议停留时长
     - 开放时段（若工具可得）

4. **路线约束规划**
   - 用高德 MCP 路径规划能力做：
     - 驾车 / 公交 / 步行 / 打车多模式估时
     - 同一天内部路线排序
     - 起终点可达性校验
     - 替代路线生成

5. **天气联动改写**
   - 用高德天气能力做：
     - 晴天优先户外
     - 雨天切换室内
     - 极端天气自动降低景点密度

6. **结构化 itinerary 输出**
   - 输出：
     - `days[]`
     - `poi[]`
     - `transport[]`
     - `weather[]`
     - `fallback_options[]`
     - `planning_trace[]`

### 5.3 必做功能增强点
尽量把高德 MCP 的价值打满，至少做这 8 个能力：

1. **真实 POI 校验**
2. **多交通方式比选**
3. **同日路线压缩与顺路排序**
4. **按酒店/机场/车站出发的动态规划**
5. **按天气改写行程**
6. **餐饮点与景点混排**
7. **周边补位**
   - 厕所、地铁口、停车场、便利店、药店
8. **可执行性评分**
   - 输出“轻松 / 正常 / 紧凑 / 不可执行”

### 5.4 新的旅游规划产品形态
前端不要只展示 Markdown,要展示:

- 地图锚点
- 每日路线卡片
- 交通耗时
- 天气提示
- 备选路线
- 可执行性评分
- 真实 POI 来源标记

## 6. 盈米金融数据 MCP 新功能设计



## **旅行资金安排助手**

### 6.1 功能定义
围绕“这次旅行的钱该怎么提前准备”，给用户一个**非投顾、非交易型**的资金规划辅助功能。

### 6.2 输入
- 旅行预算
- 出发日期
- 旅行天数
- 可接受风险级别
- 是否已有基金/现金管理持仓（可选）

### 6.3 盈米 MCP 提供的数据能力
按盈米公开描述，能力覆盖：

- 基金数据
- 投顾内容
- 投研能力
- 投顾能力
- 资金规划与资产配置引擎
- 持仓诊断与优化模型

### 6.4 输出内容
输出不要做“买哪个基金”，而做：

- 本次旅行预算占可投资资产比例
- 建议的现金预留规模
- 若资金在基金中，给出**流动性提醒**
  - T+0 / T+1 / T+N
  - 假期前赎回时点提醒
- 若预算偏高，提示对中长期资产配置的影响
- 若用户勾选“只看低波动资金安排”，只展示低风险现金管理候选分类和说明

### 6.5 展示形式
前端增加一个模块：

**“出行资金安排”**
- 总预算
- 现金预留建议
- 赎回时间提醒
- 风险提示
- 信息来源：盈米金融数据 MCP

### 6.6 合规边界
必须明确：

- 仅做信息展示与资金安排辅助
- 不构成投资建议
- 不直接做申购/赎回
- 不做收益承诺
- 不在普通旅游场景里自动触发，改为用户主动点击开启
